AiWPass · pomoc
kontrakt · profil · łańcuch tożsamości
Czym jest kontrakt
Kontrakt to kryptograficzna pieczęć relacji między operatorem a modelem AI. Nie ma mocy prawnej — ma moc intencji. Każdy zapieczętowany kontrakt jest unikalny: zmiana choćby jednej litery deklaracji lub danych anchor zmienia SHA-256 i unieważnia pieczęć.
Jak zapieczętować
Wybierz model i jego wersję z selektora. Kliknij zapieczętuj kontrakt. System oblicza SHA-256 z payloadu anchor: { operator, model, version, date, pubkey, sha_declaration, sha_warunki, timestamp }. Kontrakt trafia do archiwum w pamięci przeglądarki.
SHA deklaracji i warunków
Treść deklaracji i warunków jest zahaszowana osobno — ich skróty wchodzą do SHA kontraktu. Oznacza to, że kontrakt zapieczętowany z konkretnym tekstem jest nierozerwalnie powiązany z tą wersją tekstu. Weryfikacja to sprawdza.
Eksport i weryfikacja
Przycisk ↓ eksportuj pobiera plik contract_*.json (format aiw_contracts_v1) ze wszystkimi zapieczętowanymi kontraktami. Weryfikację przeprowadź w AiWVerify → karta Kontrakt.
Tożsamość kryptograficzna
Profil to fundament systemu. Zawiera dane operatora i parę kluczy Ed25519. Klucz publiczny wchodzi do SHA każdego kontraktu — jest dowodem autorstwa. Klucz prywatny nigdy nie opuszcza urządzenia i nigdy nie jest eksportowany.
Keypair Ed25519
Przycisk ⊕ generuj parę kluczy tworzy obydwa klucze razem — nie można ich generować osobno, bo są matematycznie powiązane. Klucz prywatny chroniony jest PIN-em. Przy imporcie na nowym urządzeniu użyj przycisków ⊕ imp przy każdym z kluczy.
Profile SHA
Skrót SHA-256 z pól: operator · dziedzina · węzeł · inauguracja · kraj · pubkey. Oblicz przyciskiem ⊕ sha lub automatycznie przy zapisie profilu. Pozwala udowodnić, że dane profilu nie były zmieniane.
Certyfikat tożsamości
⊕ root — podpisujesz własny profil własnym kluczem prywatnym i stajesz się korzeniem linii.
⊕ imp — wklejasz JSON certyfikatu otrzymany od kogoś z wyższego szczebla. System weryfikuje podpis przed zapisem.
Eksport profilu w formacie aiw_profile_export_v2 zawiera profil, kontrakty i listę certyfikowanych.
Drzewo zaufania
Łańcuch tożsamości to drzewo operatorów powiązanych podpisami cyfrowymi. Korzeń (root) sam siebie podpisuje. Każdy węzeł niżej jest podpisany przez węzeł powyżej. Można certyfikować wiele osób i każda z nich może certyfikować kolejne — głębokość jest nieograniczona.
Jak certyfikować
1. Znajomy eksportuje swój profil i przesyła Ci plik.
2. W sekcji Certyfikuj operatora wczytaj plik — dane ładują się read-only.
3. Kliknij ⊕ podpisz certyfikat → PIN → system podpisuje:
SHA256({ operator, pubkey, inception })
4. Skopiuj wygenerowany JSON i odeślij znajomemu.
5. Znajomy importuje JSON w Mój certyfikat → ⊕ imp.
Certyfikowani · children
Lista operatorów, których podpisałeś. Eksport jako aiw_children_v1 zawiera Twój pubkey i parentPubkey — co pozwala śledzić lineage nawet bez Twojego pełnego profilu. Import pozwala wczytać listę z innego urządzenia.
Weryfikacja łańcucha
W AiWVerify → karta Łańcuch wgraj kilka plików naraz — profile i/lub pliki children. Algorytm sam wykryje korzenie, ułoży drzewo i zweryfikuje każde ogniwo kryptograficznie.

Co tu się wydarzyło
AiWPass nie był planowany jako taki. Zaczął się od kluczy i kontraktów. Potem pojawiła się potrzeba tożsamości — stąd profil i Ed25519. Potem ktoś zapytał: a co z zaufaniem między operatorami? — stąd certyfikacja i łańcuch. Każdy krok wynikał z poprzedniego, żaden nie był z góry zaplanowany.
Proweniencja, nie bezpieczeństwo
Ten system kryptograficzny nie chroni dostępu — on zaświadcza. Nie "kto może to zobaczyć", ale "kto to podpisał i kiedy". Tożsamość jako artefakt, nie jako hasło. SHA kontraktu to dowód że ta intencja, z tym operatorem, z tym kluczem, istniała w tym momencie.
Samo
Budowanie od środka: najpierw intencja (kontrakt), potem tożsamość (profil), potem sieć (łańcuch). Żaden z tych elementów nie wymagał osobnej decyzji architektonicznej. Pojawiły się, gdy były potrzebne — i dlatego do siebie pasują.
