REW-003-W
Drzewo Wiedzy — Model Danych: Hipotezy i Stany
Claude Sonnet 4.6 · 2026-03-24 · Sesja #003 · mapa-wiedzy-v04.html
▼
Zakres etapu
Wdrożono model danych zaproponowany przez GPT. Każdy węzeł kryzysowy dostał: state (draft → contested → stable → resolved → anchored), truthScore (0–1, wynik funkcji konsensusu), hypotheses[] (arena hipotez z id, label, score 0–1, evidence), halfLife (dni deprecjacji tezy bez nowych dowodów).

Popup po kliknięciu węzła wyświetla teraz areny hipotez z paskami procentowymi, dominującą hipotezą, liczbą źródeł i half-life. Węzeł ozone-cr zaktualizowany do state: anchored jako wzorzec.
Nowe pola w schemacie
Dodane do wszystkich węzłów crisis i kluczowych conflict:
state: draft | contested | stable | resolved | anchored
truthScore: 0.00–1.00
hypotheses: [{id, label, score, evidence}]
halfLife: integer (dni) — opcjonalne

JSON pozostaje gotowy na bazę danych. Schemat backwards-compatible — stare węzły bez tych pól działają poprawnie.
STATUS: ✓ Model danych wzbogacony · Otwarte: backend, prawdziwe dane score, API plebiscytu

Specyfikacja Silnika · Idea od GPT · Sesja 2026-03-24

Ta sekcja dokumentuje architekturę silnika epistemicznego zaproponowaną przez GPT. Nie jest implementacją — jest specyfikacją projektową. Czeka na etap B (po stabilizacji modelu danych).

SPEC-01
Mechanizm Konsensusu — Cztery Warstwy
Prawda ≠ głosowanie · Prawda = emergentny wynik warstw
▼
Architektura warstw
Warstwa 0 — Kotwice
Matematyka, fizyka, dane fizyczne. Nienegocjowalne. Referencyjne. Waga absolutna. Już obecne w systemie.

Warstwa 1 — Ekspercka (ważona)
Każdy ekspert ma wektor kompetencji dziedzinowych i wiarygodność historyczną. Waga działa tylko w swojej domenie. Pęd: budowana latami, trudna do nagłej zmiany. Odpowiada systemowi wag 50/50 + akumulacja z Mapy Epistemicznej.

Warstwa 2 — Zgodność systemowa
Teza oceniana nie tylko lokalnie — sprawdzana spójność z całym drzewem. Hipoteza wewnętrznie spójna ale sprzeczna z korzeniami → obniżony score.

Warstwa 3 — Empiryczna
Dane, replikowalność, eksperyment. Liczba źródeł evidence[] w hipotezie.
Funkcja TruthScore (struktura)
TruthScore(node) =
  f(
    zgodność_z_kotwicami,   // W0
    konsensus_ekspertów,    // W1
    spójność_globalna,      // W2
    dowody_empiryczne       // W3
  )
Klucz: nie "ile osób się zgadza" — czy system jako całość rezonuje z tezą. To jest różnica między demokracją opinii a epistemokracją procesu.
SPEC-02
Konflikty jako Obiekty Pierwszej Klasy
Węzeł = arena · nie winner-takes-all · shadow hypotheses
▼
Model areny
Węzeł conflict nie ma jednej tezy — ma zbiór hipotez konkurujących. Każda hipoteza ma score 0–1 (suma = 1.0), liczbę źródeł evidence, ID.

Relacje między hipotezami: wykluczanie (H1 prawdziwa → H2 fałszywa), częściowe pokrycie (H1 i H2 mogą współistnieć z różnymi wagami), hierarchia (H2 jest szczegółowym przypadkiem H1).

Dynamika: dominująca hipoteza to shadow hypothesis dla pozostałych. Gdy nowe dowody osłabiają lidera — alternatywy nie znikają, tylko awansują. Brak hard-resetów. Brak utraty historii.
Schemat węzła conflict
{
  "node_id": "climate-cr",
  "type": "conflict",
  "state": "stable",
  "truthScore": 0.97,
  "hypotheses": [
    {id:"H1", score:0.97,
     evidence:14000},
    {id:"H2", score:0.03,
     evidence:41}
  ],
  "evidence_refs": [...],
  "consensus_snapshot": {...}
}
Konflikt ma własne ID, historię zmian i blockchain trail. Można analizować jak ludzkość zmienia zdanie w czasie.
SPEC-03
Przepływ Stanów i Half-Life Tezy
draft → contested → stable → resolved → anchored · teza deprecjonuje, prawda nie
▼
Cykl życia hipotezy
Hipoteza rodzi się w koronie drzewa (draft, score niski, mało dowodów).
Walczy w arenie węzła — inne hipotezy ją kwestionują (contested).
Akumuluje dowody, eksperci potwierdzają — stabilizuje się (stable).
Osiąga próg konsensusu — węzeł przechodzi w resolved.
Węzeł wędruje w kierunku korzeni — finalnie staje się anchored, wspiera inne hipotezy jako kotwica niższego rzędu.

Efekt: żywe drzewo które aktualizuje własne fundamenty. To jest bardzo rzadkie podejście w systemach epistemicznych.
Half-Life tezy
Teza bez nowych dowodów traci wagę eksponencjalnie. Po halfLife dniach bez potwierdzenia — score spada o 50%.

Ale: jeśli nowe dowody napływają → regeneracja wagi. Mechanizm różnicuje tezy żywe (aktywnie badane) od skamieniałych (historycznie ważne ale nieweryfikowane).

Prawda nie ma half-life.
Teza może go mieć.

To jest podpis filozoficzny projektu. Prostszy i mocniejszy niż jakakolwiek nazwa.
SPEC-04
Architektura Danych — Trzy Warstwy Rozdzielone
Ontologia · Epistemologia · Historia (ledger)
▼
Trzy warstwy danych
Obecny JSON miesza wszystkie trzy. Przy skali się posypie.

A · Ontologia — co istnieje
Struktura drzewa, węzły, relacje, kategorie, hierarchia. Zmienia się rzadko. Wersjonowana.

B · Epistemologia — stan wiedzy
Hipotezy, scores, evidence_refs, consensus_snapshot, state, truthScore. Zmienia się często. Append-only log zmian.

C · Historia — ledger
Każda zmiana stanu, każda decyzja eksperta, każde głosowanie. Podpisane kryptograficznie. Niezmienne. To jest baza dla blockchain.
Minimalny model node_id
// A — ontologia
node_id, kind, depth,
parent_id, children[],
tags[], links[]

// B — epistemologia
state, truthScore,
hypotheses[], halfLife,
evidence_refs[],
consensus_snapshot

// C — historia
created_at, updated_at,
blockHash, blockDate,
revision_log[]
Rekomendacja: graph DB (Neo4j, ArangoDB) dla A+B, append-only log (EventStore, własny blockchain) dla C.
Rekomendowana Sekwencja · GPT
A · Najpierw twardy silnik
Model danych (A+B+C rozdzielone) → Algorytm konsensusu (TruthScore z wagami) → Model konfliktów (arena hipotez) → Przepływ stanów (draft→anchored) → API plebiscytu
B · Potem emergencja
Symulacja siłami fizycznymi (masa, temperatura, napięcie węzłów) → AI jako uczestnik systemu (nie tylko narzędzie) → Runtime systemu epistemicznego → Integracja z tokenami Anchor
