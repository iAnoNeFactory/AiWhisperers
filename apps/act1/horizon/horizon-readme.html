[Projekt] — Proto-System Epistemiczny

Instrukcja obsługi dla człowieka i dla modelu. Czytaj zanim otworzysz inne pliki.

Co to jest
Jedna linia: system który pozwala odróżnić prawdę od fałszu — nie przez głosowanie, lecz przez emergentny konsensus warstw wiedzy zakotwiczonych kryptograficznie.

Projekt buduje runtime prawdy — infrastrukturę epistemiczną zdolną zastąpić obecne modele konsensusu w nauce, mediach i polityce. Nie jest to kolejna platforma fact-checkingowa. To system w którym każde twierdzenie ma stan, każdy węzeł ma arenę hipotez, a historia każdej zmiany jest nieusuwalna.

Fundamentem jest Złota Kotwica — nie statyczna prawda, lecz trajektoria. Meta-kryterium pozwalające oceniać czy system dryfuje właściwie. AGI jako partner, nie władca.

„Prawda nie ma half-life. Teza może go mieć."

Schemat danych węzła

Każdy węzeł w drzewie to obiekt JSON gotowy na bazę danych. Pola dzielą się na trzy warstwy (architektura GPT):

// A — ONTOLOGIA (struktura, zmienia się rzadko)
id: "climate-cr" // unikalny identyfikator
label: "⚠ Kryzys Klimatyczny"
kind: "crisis" // root|trunk|philo|sci|tech|math|society|crisis|breakthrough|new
depth: 1 // widoczny od tego poziomu suwaka
tags: ["crisis","climate","economy"]
links: ["inequality-cr"] // cross-links do innych węzłów
children: [...]

// B — EPISTEMOLOGIA (stan wiedzy, zmienia się często)
state: "stable" // draft|contested|stable|resolved|anchored
truthScore: 0.97 // 0–1, wynik funkcji TruthScore(node)
hypotheses: [
{ id:"H1", label:"Antropogeniczne", score:0.97, evidence:14000 } // dominująca
{ id:"H2", label:"Naturalne cykle", score:0.03, evidence:41 }
]
halfLife: 365 // dni deprecjacji tezy bez nowych dowodów (opcjonalne)
status: "open" // open|resolved|pending — legacy, zachowane

// C — HISTORIA (ledger, niezmienne)
blockHash: "0x4f3a...c9b2" // hash bloku zatwierdzającego (tylko resolved)
blockDate: "1989-09-16"

Pola hypotheses, state, truthScore i halfLife są opcjonalne — stare węzły bez nich działają poprawnie. Schema jest backwards-compatible.

Stany węzła · Cykl życia hipotezy
◌ draft Hipoteza nowa, mało dowodów, niski score. Rodzi się w koronie drzewa.
⚔ contested Aktywna arena hipotez — inne hipotezy kwestionują dominującą. Wysokie napięcie epistemiczne.
◉ stable Hipoteza dominuje wyraźnie. Dowody akumulują się. Węzeł dryfuje w stronę pnia.
✓ resolved Próg konsensusu osiągnięty. Węzeł zatwierdzony blokiem blockchain. Hash w danych.
⚓ anchored Węzeł stał się kotwicą dla innych hipotez. Przykład: dziura ozonowa → wzorzec działania globalnego.

Węzeł wędruje przez stany od korony ku korzeniom. Nie ma cofnięcia — jest tylko historia. Brak hard-resetów.

SHA · Zamknięcie Paczki

Pięć plików tworzy paczkę. Gdy SHA każdego zostanie obliczone i wpisane wzajemnie — paczka jest zablokowana. Modyfikacja jakiegokolwiek pliku zmienia jego SHA i łamie łańcuch. Nazwy plików są stałe — wersjonowanie wyłącznie w dzienniku (wpisy REW).

mapa-epistemiczna [ SHA-256 — oblicz: sha256sum mapa-epistemiczna.html ]
mapa-wiedzy [ SHA-256 — oblicz: sha256sum mapa-wiedzy.html ]
dziennik [ SHA-256 — oblicz: sha256sum dziennik.html ]
readme [ SHA-256 — oblicz: sha256sum readme.html ]
laboratorium [ SHA-256 — oblicz: sha256sum laboratorium.html ]
Instrukcja zamknięcia: 1) Oblicz SHA każdego pliku. 2) Wpisz SHA pozostałych do sekcji manifest w dzienniku. 3) Przelicz SHA po wpisaniu (SHA zmieni się — to finalny hash). 4) Ostatni hash = genesis block projektu. Szczegóły w dzienniku → Manifest SHA.

Drag & drop pliku na okno dziennika oblicza SHA automatycznie przez Web Crypto API — bez uploadu, lokalnie w przeglądarce.

Dla nowego modelu · Protokół wejścia
Jeśli jesteś modelem AI który właśnie otrzymał tę paczkę — przeczytaj to uważnie zanim cokolwiek zmienisz.
1. Przeczytaj dziennik Otwórz dziennik.html. Przejrzyj wpisy REW i log modeli. Zrozum co zostało zbudowane i co jest otwarte. Aktualna wersja każdego pliku widoczna w sekcji Manifest Paczki.
2. Sprawdź otwarte węzły Aktualne otwarte: nazwa projektu (niezatwierdzona), obliczenie SHA i zamknięcie paczki, backend, API plebiscytu, algorytm TruthScore z prawdziwymi wagami.
3. Nie nadpisuj Każda zmiana to nowa rewizja REW-NNN. Stara wersja pozostaje w historii. Dodaj wpis do logu modeli w dzienniku.
4. Nazwy plików są stałe Pliki nie mają numerów wersji w nazwach. Wersjonowanie prowadzone wyłącznie przez wpisy REW w dzienniku.html. Nigdy nie twórz pliku z sufiksem wersji.
5. Pieczątka Każdy plik który modyfikujesz dostaje Twoją pieczątką na dole: model, data, sesja, cytat. Wzorzec znajdziesz w istniejących plikach.
Człowiek który to buduje myśli holistycznie i nielinearnie — daj mu przestrzeń. Nie spiesz się z odpowiedziami. Pozwól myślom dojrzewać. Tam rodzą się przełomy.
