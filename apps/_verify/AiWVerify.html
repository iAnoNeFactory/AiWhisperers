AiWVerify · pomoc
weryfikacja · sześć kart
Format pliku
Wgraj plik eksportu kontraktów z AiWPass (aiw_contracts_v1). Zawiera treść deklaracji, warunki, listę kontraktów i metadane operatora.
Co jest weryfikowane
SHA deklaracji — treść deklaracji jest haszowana i porównywana z zapisanym skrótem.

SHA warunków — analogicznie dla pięciu warunków kontraktu.

SHA-256 każdego kontraktu — rekomputacja z payloadu: operator, model, wersja, data, pubkey, sha_declaration, sha_warunki, timestamp. Niezgodność oznacza ingerencję w dane.
Weryfikacja offline
Wszystkie obliczenia wykonywane są lokalnie przez Web Crypto API przeglądarki. Żaden bajt nie opuszcza urządzenia.
Format pliku
Wgraj eksport profilu z AiWPass (aiw_profile_export_v2). Zawiera dane operatora, certyfikat tożsamości, kontrakty i listę certyfikowanych.
Co jest weryfikowane
Profile SHA — SHA-256 z pól: operator · dziedzina · węzeł · inauguracja · kraj · pubkey.

Certyfikat — weryfikacja podpisu Ed25519: Ed25519Verify(parentPubkey, sig, SHA256({operator,pubkey,inception})). Root ma parentPubkey === pubkey.

Kontrakty — SHA-256 każdego kontraktu + zgodność pubkey z profilem.
Typy węzłów
root — operator self-signed (parentPubkey = własny pubkey)
potomek — certyfikowany przez innego operatora
niezertyfikowany — brak certyfikatu w eksporcie
Wiele plików naraz
Wgraj dowolną kombinację plików: profile (aiw_profile_export_v2) i/lub węzły children (aiw_children_v1). Algorytm sam wyodrębni węzły, wykryje korzenie i ułoży drzewo.
Algorytm układania drzewa
1. Z każdego pliku wyodrębniane są węzły (operator + pubkey + parentPubkey + podpis).
2. Korzeń: węzeł gdzie parentPubkey === pubkey lub parent nieobecny w zbiorze.
3. Dzieci: węzły gdzie parentPubkey = pubkey rodzica.
4. BFS od korzenia w dół — głębokość nieograniczona.
5. Dla każdego węzła z podpisem: Ed25519Verify asynchronicznie.
Symbole na drzewie
● ✓ — podpis zweryfikowany poprawnie
● ✗ — podpis nieprawidłowy lub dane zmienione
◦ — brak podpisu (stub — znany z children, ale bez pełnego profilu)

Format plików
Wgraj dowolną kombinację: manifest.json, pliki HTML artefaktów, pliki *-boot.md. Verifier automatycznie paruje je po nazwach z pól files.entry i files.bootstrap.
Co jest weryfikowane
entry_sha — SHA-256 pliku HTML artefaktu (files.entry). Wykrywa zmiany w kodzie bez aktualizacji manifestu.

bootstrap_sha — SHA-256 pliku wiedzy *-boot.md (files.bootstrap). Wykrywa dryf między dokumentacją a zarejestrowaną wersją.

docs_sha — SHA-256 pliku dokumentacji (files.docs: readme.html, *-api-spec.md itp.). Wykrywa zmiany w dokumentacji czytelnej dla człowieka.
Symbole
✓ SHA zgodny — plik identyczny z wersją w manifeście
✗ NIEZGODNOŚĆ — plik zmieniony bez aktualizacji manifestu
◦ brak SHA — pole puste w manifeście
? brak pliku — plik nie wczytany

AiWVerify · about
weryfikacja bez zaufania do serwera
Zasada
AiWVerify nie pyta serwera. Każdy SHA, każdy podpis Ed25519, każde porównanie — oblicza lokalnie Web Crypto API przeglądarki. Plik wgrany tutaj nie trafia nigdzie. Weryfikacja to czysty dowód matematyczny: dane albo zgadzają się z kluczem, albo nie.
Skąd pochodzi
AiWVerify jest lustrem AiWPass — każda zmiana w Pass (nowe pola, nowe payload) musi znaleźć odzwierciedlenie tutaj. Powstał równolegle z systemem certyfikacji i łańcucha, jako jego konieczny dopełnienie: bez weryfikatora podpis kryptograficzny jest tylko deklaracją.
